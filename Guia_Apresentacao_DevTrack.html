🎤 Roteiro de Apresentação: DevTrack

Este é o guia definitivo para a sua apresentação de 15 minutos, dividido exatamente entre você (Fullstack) e seus amigos (Banco de Dados).

👨‍💻 PARTE 1: Você (Frontend e Backend) - Aprox. 8 a 10 minutos

Sua missão é mostrar que o aplicativo é real, rápido, seguro e construído com as melhores práticas de mercado.

1. Abertura e O Problema (1 min)

"Bom dia/boa noite a todos. Nosso projeto é o DevTrack. Percebemos que as plataformas de estudo hoje focam muito na validação social. Nós criamos uma solução focada na disciplina individual. Inspirados na gamificação do Duolingo, transformamos o estudo diário de programação em um jogo, onde o usuário ganha XP, mantém ofensivas (streaks) e compete num ranking local."

2. O Frontend: Mobile e UI (3 min)

"No Frontend, construímos o aplicativo usando React Native com Expo e TypeScript. O grande diferencial visual do nosso app foi o uso da biblioteca Reanimated. Todas as animações, como o carregamento do XP e as transições de tela, rodam a 60 quadros por segundo na thread nativa do aparelho, garantindo uma fluidez premium. Para navegação, usamos o Expo Router e para o estado global aplicamos a Context API do React."

3. O Backend: Regras de Negócio e Segurança (4 min)

"Toda a inteligência do DevTrack mora no nosso Backend, desenvolvido em Java 21 com Spring Boot 3. Nós estruturamos o código seguindo a Clean Architecture (Arquitetura Limpa), separando rotas, regras de negócio e banco de dados.

Para segurança, implementamos o Spring Security com JWT (JSON Web Token). Nossa API é 100% stateless. Se um 'Visitante' tentar acessar uma área logada, o Spring Boot barra a requisição devolvendo um Erro 403 (Proibido), garantindo a segurança dos dados."

🗄️ PARTE 2: Seus Amigos (Banco de Dados) - Aprox. 5 a 7 minutos

Fala 1: O Banco de Dados Central (MySQL) - Amigo 1

"Para suportar toda a lógica do servidor, optamos pelo MySQL. Nós não criamos as tabelas manualmente. Utilizamos o Hibernate (JPA) integrado ao Spring Boot. Isso significa que nosso banco é gerado a partir do código Java (Code-First). O uso do JPA garante que todas as nossas queries sejam otimizadas e protegidas nativamente contra ataques de SQL Injection."

Fala 2: A Estratégia Offline-First (SQLite) - Amigo 2

"Porém, estamos lidando com um aplicativo Mobile, e conexões caem. Para resolver isso, implementamos uma arquitetura Offline-First. No celular do usuário, rodamos um banco de dados local usando o SQLite e o AsyncStorage. Sempre que há internet, sincronizamos. Se o usuário abrir o app no modo avião, ele lê os dados do SQLite instantaneamente. Assim, funcionalidades como o Pomodoro e Vidas funcionam 100% sem internet."

🎯 PARTE 3: A Banca (Perguntas e Respostas)

❓ "Por que usar JWT em vez de Sessão comum (Cookies)?"

Você: "Como temos Frontend Mobile e Backend separados, o JWT permite que a API seja stateless. O servidor não gasta memória lembrando quem está logado; a autorização viaja criptografada dentro do token na requisição."

❓ "Como vocês garantem que a senha do usuário não vai ser roubada no banco?"

Amigos (BD): "No MySQL, nunca salvamos a senha em texto puro. Utilizamos o algoritmo de hash BCrypt. Ele gera um código irreversível, nem nós devs sabemos as senhas."

❓ "E se houver milhares de usuários, o Ranking não vai deixar o servidor lento?"

Amigos (BD): "Nossa consulta já utiliza paginação nativa do Spring Data. E num cenário real, adicionaríamos Índices (Indexes) na coluna XP e Streak no MySQL para que a leitura seja em frações de segundo."
